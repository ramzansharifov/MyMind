Модуль заметок MyMind: архитектура, библиотеки, решения и проблемы
=================================================================

1. Общая идея модуля
--------------------

Модуль заметок в MyMind устроен как локальный block-based редактор. По смыслу он ближе к Notion/Anytype, чем к обычному textarea: заметка состоит не из одной большой строки текста, а из набора блоков.

Каждый блок имеет свой тип:

- обычный текст;
- заголовок;
- список;
- чек-лист;
- toggle-блок;
- таблица;
- изображение;
- видео;
- аудио;
- файл;
- код;
- markdown-блок;
- разделитель;
- доска для рисования.

Главная цель архитектуры: дать пользователю возможность собирать заметку как документ из отдельных смысловых частей, но при этом сохранить простой локальный формат хранения, чтобы всё лежало в JSON и не требовало сервера.


2. Основные файлы модуля
------------------------

Ключевые файлы:

- `src/modules/notes/NotesPage.tsx`
  Страница списка заметок. Здесь отображаются карточки заметок, группы, поиск, фильтры, создание и открытие заметки.

- `src/modules/notes/NoteEditorPage.tsx`
  Главная страница редактора заметки. Здесь живёт BlockNote-редактор, верхняя панель, режим чтения, режим визуального редактора, правая панель настроек блока, сохранение, guard от выхода с несохранёнными изменениями.

- `src/modules/notes/NoteCard.tsx`
  Карточка заметки в списке.

- `src/modules/notes/types.ts`
  Типы данных заметок: `Note`, `NotesData`, свойства заметок, assets, ширина документа.

- `src/modules/notes/noteUtils.ts`
  Утилиты нормализации, миграции старого контента, извлечения plain text, конвертации legacy-структур.

- `src/modules/notes/editor/noteSchema.ts`
  Кастомная схема BlockNote. Здесь подключены стандартные блоки BlockNote и наш кастомный drawing-блок.

- `src/modules/notes/editor/contentSanitizer.ts`
  Санитайзер входящего editorContent. Он защищает редактор от неизвестных или старых блоков, подготавливает drawing-блоки, а также отключает тяжёлые preview в визуальном редакторе.

- `src/modules/notes/editor/blockActions.ts`
  Набор низкоуровневых операций над блоками: поиск текущего блока, вставка, flatten, подсчёт визуальных блоков, работа со списками.

- `src/modules/notes/editor/ReadOnlyBlocks.tsx`
  Собственный рендер заметки в режиме чтения. Мы не используем BlockNote как основной read-only renderer, потому что нам нужен более контролируемый внешний вид.

- `src/modules/notes/editor/constants.ts`
  Константы поддерживаемых типов блоков, цветов, списков, drawing-настроек.

- `src/modules/notes/blocks/drawing/*`
  Кастомный блок для рисования.

- `src/styles/modules/notes.css`
  Главный файл стилей заметок. Он импортирует более мелкие CSS-файлы.

- `src/styles/modules/notes-core.css`
  Основные стили страницы заметок, редактора, режима чтения и кастомизации BlockNote.

- `src/styles/modules/notes-editor-overrides.css`
  Узкие override-стили для поведения BlockNote, особенно для медиа и отображения переносов.


3. Какие библиотеки используются
--------------------------------

3.1. BlockNote

Главная библиотека редактора:

- `@blocknote/core`
- `@blocknote/react`
- `@blocknote/mantine`

BlockNote даёт:

- block-based модель документа;
- стандартные блоки;
- slash menu;
- side menu;
- таблицы;
- file/image/video/audio blocks;
- inline-форматирование;
- сериализацию документа;
- API `editor.document`, `editor.updateBlock`, `editor.insertBlocks`, `editor.blocksToHTMLLossy`.

Почему выбрали BlockNote:

- не нужно писать ProseMirror/Tiptap-редактор с нуля;
- есть готовая структура блоков;
- легко расширять кастомными блоками;
- можно хранить документ как JSON-массив блоков;
- библиотека уже решает большую часть сложной логики курсора, выделения, таблиц, списка, inline marks.

Какие ограничения BlockNote проявились:

- стандартный file/image/video preview может быть тяжёлым;
- некоторые визуальные элементы библиотеки пришлось переопределять CSS-ом;
- часть поведения списков и toggle-блоков пришлось адаптировать под наш UX;
- стандартный read-only режим не дал нужного визуального контроля, поэтому мы сделали свой renderer для чтения.


3.2. Lucide React

Иконки:

- `lucide-react`

Используются в:

- sidebar;
- верхней панели редактора;
- кнопках блоков;
- правой панели настроек;
- карточках;
- модальных окнах.


3.3. React

Весь интерфейс построен на React:

- страницы модулей;
- редактор заметок;
- right sidebar;
- модальные окна;
- карточки;
- группы;
- настройки.


3.4. Electron

Приложение desktop-first. Electron нужен для:

- локального хранения JSON;
- доступа к папке данных;
- открытия системных папок;
- сохранения медиа-файлов заметок в локальную папку assets;
- работы без сервера.

Сейчас preload (`electron/preload.ts`) отдаёт renderer-части API:

- `window.mymind.storage`;
- `window.mymind.files`.

Через `window.mymind.files` заметки теперь могут сохранять вложения в локальную assets-папку.


4. Формат данных заметки
------------------------

Основной тип находится в `src/modules/notes/types.ts`.

Заметка содержит:

- `id`;
- `title`;
- `content`;
- `contentFormat`;
- `category`;
- `groupId`;
- `tags`;
- `pinned`;
- `editorContent`;
- `editorPlainText`;
- `editorHtml`;
- `properties`;
- `assets`;
- `schemaVersion`;
- `layoutWidth`.

Главное поле для современного редактора:

- `editorContent`.

Это массив BlockNote-блоков. Примерно:

```json
[
  {
    "id": "...",
    "type": "paragraph",
    "content": "Текст заметки",
    "props": {}
  },
  {
    "id": "...",
    "type": "image",
    "props": {
      "url": "file:///C:/Users/.../MyMind/data/assets/image.png",
      "showPreview": false
    }
  }
]
```

`content`, `editorPlainText` и `editorHtml` нужны для:

- поиска;
- превью карточек;
- обратной совместимости;
- простого отображения там, где не нужен полный документ.


5. Хранение данных
------------------

Данные заметок лежат локально в JSON:

- `Documents/MyMind/data/notes.json`

Общий storage-слой находится в:

- `src/shared/app/useAppData.ts`
- `src/shared/storage/storageClient.ts`
- `electron/ipc/storage.ipc.ts`

Раньше медиа-файлы могли попадать прямо в JSON как `data:base64`. Это работало для маленьких картинок, но плохо подходило для видео.

Сейчас для новых файлов добавлена более правильная схема:

- файл один раз читается при добавлении;
- сохраняется в `Documents/MyMind/data/assets`;
- в BlockNote-документ кладётся короткий `file://` URL;
- визуальный редактор показывает лёгкую заглушку;
- режим чтения загружает реальное медиа.

Это важно, потому что JSON с base64-видео может стать огромным и начать тормозить весь редактор.


6. Режимы редактора
-------------------

У заметки есть два основных режима:

- режим чтения;
- визуальный редактор.

Отдельный markdown-режим был убран как самостоятельный режим. Вместо него есть markdown/code block.


6.1. Визуальный редактор

Визуальный редактор использует `BlockNoteView`.

Файл:

- `src/modules/notes/NoteEditorPage.tsx`

Основные настройки:

- `schema: noteSchema`;
- `initialContent`;
- `uploadFile`;
- `formattingToolbar={false}`;
- `slashMenu={false}`;
- `sideMenu={false}`;
- кастомный slash menu;
- кастомный side menu;
- кастомная правая панель настроек.

Мы отключили стандартные части UI BlockNote там, где они мешали нашему дизайну, и вынесли управление в собственные элементы.

Визуально редактор устроен так:

- сверху находится панель действий: назад, режим чтения, визуальный редактор, ширина документа, сохранить;
- ниже находится заголовок заметки;
- ниже метаданные и теги;
- ниже быстрая панель добавления блоков;
- основная зона документа;
- справа панель настроек выбранного блока.

Каждый блок выглядит как отдельная карточка/секция. Это было сделано специально: пользователь собирает заметку как конструктор из независимых блоков.


6.2. Режим чтения

Режим чтения рендерится через `ReadOnlyBlocks`, а не через обычный BlockNote editor.

Причины:

- нужен более чистый вид;
- не нужны рамки редактирования;
- не нужны handles, side menu, placeholders;
- toggle должен раскрывать/скрывать дочерний контент;
- списки должны визуально объединяться;
- drawing должен показываться как статичная доска;
- медиа должны показываться полноценно только здесь.

В режиме чтения документ выглядит как единая страница, а не набор активных editor-блоков. При этом сохраняются внешние отступы и визуальное разделение смысловых частей.


7. Кастомная BlockNote-схема
----------------------------

Файл:

- `src/modules/notes/editor/noteSchema.ts`

Мы берём стандартные BlockNote block specs:

- `defaultBlockSpecs`

И добавляем:

- наш `drawing` block.

Также мы меняли поведение Enter:

- для большинства inline-блоков Enter вставляет перенос строки внутри блока;
- для списков и toggle используется библиотечное поведение, потому что там Enter должен создавать следующий пункт списка или работать с вложенной структурой.

Это решение появилось из-за проблемы:

- пользователь хотел, чтобы обычный paragraph не распадался на новый блок при каждом Enter;
- но списки должны оставаться списками, а не превращаться в многострочный plain text.


8. Поддерживаемые блоки
-----------------------

Список поддерживаемых блоков хранится в:

- `src/modules/notes/editor/constants.ts`

Сейчас поддерживаются:

- `paragraph`;
- `heading`;
- `quote`;
- `bulletListItem`;
- `numberedListItem`;
- `checkListItem`;
- `toggleListItem`;
- `divider`;
- `table`;

Если при открытии заметки в `editorContent` попадает неизвестный блок, sanitizer превращает его в paragraph, чтобы редактор не падал.


9. Правая панель настроек блока
-------------------------------

Правая панель находится в `NoteEditorPage.tsx`.

Она показывает настройки выбранного блока:

- тип блока;
- тип маркера списка;
- тип toggle;
- форматирование;
- ссылка;
- цвет текста;
- цвет фона;
- ширина картинки;
- caption;
- язык code/markdown block;
- настройки drawing-блока;
- дублирование;
- удаление.

Важная UX-идея:

Раньше часть настроек открывалась через меню шести точек BlockNote. Мы перенесли эту логику в правый sidebar, чтобы настройки были стабильными, видимыми и соответствовали дизайну приложения.


10. Быстрая панель блоков
-------------------------

Визуальный редактор имеет верхнюю панель быстрых блоков:

- Text;
- List;
- Table;
- Image;
- Code;
- More blocks.

В More blocks находятся:

- Markdown;
- Quote;
- Toggle;
- Divider;
- Drawing;
- Video;
- Audio;
- File.

Заголовки не вынесены отдельными кнопками, потому что heading можно выбрать в настройках обычного текстового блока.


11. Списки
----------

Со списками было несколько важных проблем.

Проблема 1:

BlockNote по умолчанию создаёт каждый пункт списка как отдельный block.

Это нормально для внутренней модели BlockNote, но визуально пользователь хотел воспринимать список как один общий блок.

Что сделано:

- в визуальном редакторе соседние list-блоки одного типа визуально группируются CSS-ом;
- в режиме чтения `ReadOnlyBlocks` собирает соседние list-блоки в один `<ul>` или `<ol>`;
- в статусбаре такие соседние пункты считаются как один визуальный блок.

Проблема 2:

Нужно было иметь один тип `List` в UI, а уже в правой панели выбирать marker type.

Что сделано:

- кнопка `List` создаёт список;
- в правой панели можно выбрать marker type;
- при смене marker type меняется группа соседних list-блоков.

Проблема 3:

Checkbox-list выглядел не в стиле приложения.

Что сделано:

- чекбоксы стилизованы под кастомные чекбоксы MyMind;
- в режиме чтения и редактирования внешний вид приведён к одному стилю;
- убрано зачёркивание текста при галочке в режиме редактирования.


12. Toggle-блоки
----------------

Toggle-блоки тоже были доработаны.

Изначально были отдельные варианты:

- toggle list;
- heading toggle.

Мы объединили это в один UI-тип `Toggle`.

В правой панели можно выбрать:

- обычный toggle;
- heading toggle 1;
- heading toggle 2;
- heading toggle 3.

В режиме чтения toggle раскрывается и скрывается через состояние внутри `ReadOnlyBlocks`.

Визуальные доработки:

- стрелка получила акцентный цвет;
- убрана лишняя рамка;
- стрелка выровнена относительно родительского текста;
- дочерний контент не должен сбивать позиционирование стрелки.


13. Markdown
------------

Раньше был отдельный markdown-режим заметки.

Сейчас отдельного режима нет. Вместо него используется markdown/code block:

- пользователь добавляет Markdown block;
- по сути это `codeBlock` с `language: markdown`;
- в режиме чтения markdown может быть отрендерен как HTML через `markdownToHtml`.

Почему так лучше:

- не нужно поддерживать два режима редактирования одной заметки;
- markdown становится частью block-документа;
- можно смешивать обычный текст, таблицы, drawing и markdown в одной заметке.


14. Drawing-блок
----------------

Drawing-блок находится в:

- `src/modules/notes/blocks/drawing`

Идея:

- простая доска для рисования внутри заметки;
- без полноценной whiteboard-системы;
- без Excalidraw;
- без zoom/pan;
- без фигур и сложных инструментов.

Функции:

- рисование мышкой/тачпадом/стилусом;
- очистка;
- undo;
- redo;
- цвет кисти;
- толщина кисти;
- высота canvas в пикселях.

Данные drawing-блока:

- во время работы держатся отдельно через внутренний store drawing-блока;
- при сохранении `mergeDrawingBlockData` добавляет `drawingData` обратно в блок;
- в editorContent хранится base64-изображение рисунка.

Почему drawing хранится иначе, чем видео:

- рисунок обычно гораздо меньше видео;
- это собственный пользовательский контент;
- его нельзя восстановить из локального файла;
- base64 для drawing приемлем как компромисс.

Проблемы, с которыми сталкивались:

- блок перерисовывался после каждого штриха;
- из-за этого ломались undo/redo и пропадал sidebar;
- решение: не обновлять весь BlockNote-блок на каждый stroke, а аккуратно держать drawing data отдельно и сливать при сохранении.


15. Медиа: изображения, видео, аудио, файлы
------------------------------------------

Это самая важная часть текущей оптимизации.

15.1. Как было раньше

BlockNote вызывает наш `uploadFile(file)`.

Раньше `uploadFile` делал так:

```ts
reader.readAsDataURL(file)
```

То есть любой файл превращался в `data:base64`.

Для маленьких картинок это терпимо.

Для видео это плохо:

- видео в 100 МБ превращается в ещё больший base64;
- base64 попадает прямо в `notes.json`;
- `editorContent` становится огромным;
- каждый ввод в редакторе начинает работать с огромной строкой;
- JSON дольше читается и пишется;
- React/BlockNote вынуждены держать тяжёлый документ в памяти;
- визуальный редактор начинает лагать.

15.2. Что сделано сейчас

Для новых файлов используется новая схема:

1. Пользователь добавляет файл в BlockNote.
2. `uploadFile(file)` пытается сохранить файл как локальный asset.
3. Electron IPC пишет файл в:

   `Documents/MyMind/data/assets`

4. В `editorContent` сохраняется только короткая ссылка:

   `file:///.../Documents/MyMind/data/assets/...`

5. В визуальном редакторе `showPreview` ставится в `false`.
6. В режиме чтения реальный файл показывается:

   - image как `<img>`;
   - video как `<video controls>`;
   - audio как `<audio controls>`;
   - file как ссылка.

15.3. Почему в редакторе теперь заглушка для любых файлов

Пользователь попросил не только видео, но и фото показывать в редакторе как лёгкую заглушку.

Теперь lightweight-preview применяется к:

- image;
- video;
- audio;
- file.

То есть визуальный редактор не должен грузить тяжёлое медиа как полноценный preview. Он показывает компактный блок-заглушку/карточку файла, а реальный просмотр происходит в режиме чтения.

Это уменьшает:

- расход памяти;
- количество DOM-узлов с тяжёлым media;
- вероятность лагов при вводе;
- риск зависания при открытии большой заметки.

15.4. Что будет со старыми заметками

Если старое видео уже было сохранено как base64 в `notes.json`, мы не можем автоматически восстановить путь к исходному файлу. В base64 нет информации о том, откуда файл был взят.

Поэтому старое тяжёлое видео лучше:

1. удалить из заметки;
2. добавить заново.

После повторного добавления оно попадёт в `data/assets`, а в заметке останется лёгкая ссылка.


16. Режим чтения медиа
----------------------

Режим чтения должен быть красивым и полноценным.

Поэтому:

- изображения показываются реальной картинкой;
- видео показывается плеером;
- аудио показывается audio player;
- файл показывается ссылкой;
- drawing показывается как изображение/доска;
- таблицы показываются как таблицы;
- списки объединяются визуально.

Важно:

`showPreview: false` нужен только для визуального редактора. Режим чтения не должен считать это запретом на просмотр.


17. Санитайзер контента
-----------------------

Файл:

- `src/modules/notes/editor/contentSanitizer.ts`

Задачи:

- проверить, что `editorContent` является массивом;
- удалить или преобразовать неизвестные блоки;
- сохранить только поддерживаемые блоки;
- подготовить drawing-блоки;
- отключить preview для тяжёлых media-блоков в редакторе;
- создать пустой блок нужного типа при добавлении через UI.

Почему это важно:

BlockNote может падать, если ему передать документ с неизвестным блоком или некорректными props. Санитайзер — это слой защиты между локальным JSON и редактором.


18. Сохранение заметок
----------------------

Сохранение происходит вручную по кнопке.

Мы убрали автосохранение на каждое изменение, потому что:

- при больших документах это вызывает лаги;
- медиа-блоки увеличивают стоимость сериализации;
- пользователь должен спокойно писать без микрофризов.

При сохранении:

1. drawing data сливается обратно в блоки через `mergeDrawingBlockData`.
2. plain text извлекается через `editorContentToPlainText`.
3. HTML создаётся через `editor.blocksToHTMLLossy`.
4. Собирается объект `Note`.
5. Он передаётся в общий app data layer.
6. Данные пишутся в JSON.

Если пользователь пытается выйти с несохранёнными изменениями, показывается модальное окно:

- остаться;
- выйти без сохранения;
- сохранить и выйти.

Guard работает не только для кнопки "Назад к заметкам", но и при попытке перейти в другой модуль.


19. Визуальный дизайн
---------------------

Заметки следуют общему стилю MyMind:

- тёмная тема;
- glass-like контейнеры;
- сине-тёмные поверхности;
- акцентный цвет;
- мягкие границы;
- restrained hover-состояния;
- компактные кнопки;
- right sidebar для настроек;
- модальные окна по центру экрана;
- общий blur-overlay для модальных окон.

Визуальный редактор:

- блоки выглядят как отдельные карточки;
- у каждого блока есть свой фон и рамка;
- кнопки плюсика и шести точек появляются как side menu;
- настройки блока вынесены в правую панель;
- медиа в редакторе отображаются как лёгкие карточки, а не как полноценные плееры.

Режим чтения:

- выглядит как единый документ;
- блоки не выглядят редактируемыми;
- нет лишних рамок редактора;
- toggle интерактивен;
- медиа показывается полноценно;
- таблицы и списки выглядят как часть документа.


20. Почему не используем полноценную whiteboard-библиотеку
---------------------------------------------------------

Изначально рассматривался Excalidraw.

От него отказались, потому что задача была не в полноценной доске, а в простом drawing-блоке внутри заметки.

Excalidraw дал бы:

- больше возможностей;
- больше веса;
- больше состояния;
- больше сложной интеграции;
- отдельную модель документа.

Для MyMind сейчас лучше простой canvas:

- быстрее;
- легче;
- проще поддерживать;
- достаточно для быстрых зарисовок.


21. Главные проблемы, с которыми столкнулись
-------------------------------------------

21.1. Enter в обычном тексте

Проблема:

Enter создавал новый блок, а пользователь ожидал перенос строки внутри paragraph.

Решение:

- для обычных inline-блоков добавлен hard break по Enter;
- для списков и toggle оставлено библиотечное поведение.


21.2. Списки выглядят как отдельные блоки

Проблема:

Внутри BlockNote каждый пункт списка — отдельный block.

Решение:

- визуально группируем соседние list-блоки;
- в режиме чтения собираем их в один список;
- в sidebar меняем настройки сразу для группы.


21.3. Toggle в режиме чтения

Проблема:

Read-only BlockNote не давал нужного поведения.

Решение:

- собственный renderer `ReadOnlyBlocks`;
- состояние раскрытия хранится в React;
- стрелка и дочерний контент стилизованы вручную.


21.4. Drawing-блок перерисовывался

Проблема:

Если сохранять drawing data в BlockNote props на каждый stroke, весь блок перерисовывается.

Решение:

- не обновлять BlockNote-блок на каждый stroke;
- хранить drawing data отдельно во время редактирования;
- сливать данные в editorContent только при сохранении.


21.5. Видео тормозит редактор

Проблема:

Видео сохранялось как base64 внутри JSON.

Решение:

- медиа сохраняется в локальную папку `assets`;
- в документе хранится лёгкий URL;
- в редакторе показывается заглушка;
- в режиме чтения показывается реальный плеер.


21.6. Старые данные

Проблема:

У приложения уже были старые заметки и старый формат контента.

Решение:

- `noteUtils.ts` умеет мигрировать старые блоки;
- `sanitizeInitialContent` защищает редактор от плохого входа;
- для новых архитектурных изменений стараемся делать совместимые преобразования.


22. Как сейчас добавить новый тип блока
--------------------------------------

Чтобы добавить новый блок:

1. Добавить тип в `SUPPORTED_BLOCK_TYPES`.
2. Если это кастомный BlockNote-блок, добавить block spec в `noteSchema.ts`.
3. Добавить создание пустого блока в `createEmptyBlock`.
4. Добавить кнопку в `QuickBlockToolbar` или `More blocks`.
5. Если нужен slash menu, добавить item в `CustomSlashMenu`.
6. Добавить настройки в правый sidebar `NotePropertiesPanel`.
7. Добавить read-only render в `ReadOnlyBlocks`.
8. Добавить стили в CSS.
9. Убедиться, что `editorContentToPlainText` корректно извлекает текст/описание.
10. Проверить, что sanitizer не удаляет новый блок.


23. Как сейчас добавить новую настройку блока
---------------------------------------------

Обычно настройка блока делается так:

1. Получить текущий блок через `selectedBlock`.
2. Показать контрол в `NotePropertiesPanel`.
3. Изменить props через `editor.updateBlock`.
4. Вызвать `onDirty`.
5. При необходимости обновить read-only renderer.
6. При необходимости добавить CSS.

Пример:

- image width хранится в `props.previewWidth`;
- drawing height хранится в `props.canvasHeight`;
- marker type списка фактически меняет тип соседних list-блоков;
- toggle type меняет `toggleListItem` или heading с `isToggleable`.


24. Что ещё можно улучшить позже
--------------------------------

24.1. Asset manager

Сейчас assets сохраняются локально, но пока нет полноценного UI для управления ими.

Позже можно добавить:

- список вложений заметки;
- удаление неиспользуемых assets;
- проверку битых ссылок;
- перенос assets при экспорте backup-файла;
- отображение размера файла.


24.2. Очистка старого base64

Можно сделать миграцию:

- найти data:video/audio/image в старых заметках;
- предложить пользователю экспортировать их в assets;
- заменить base64 на file:// URL.

Но это нужно делать аккуратно, потому что большие base64-строки могут быть тяжёлыми для обработки.


24.3. Более умный preview

Можно сделать отдельный собственный media placeholder:

- иконка типа файла;
- имя файла;
- размер;
- кнопка открыть;
- кнопка заменить файл;
- кнопка показать preview временно.

Сейчас мы используем lightweight-поведение BlockNote через `showPreview: false`.


24.4. Виртуализация больших документов

Если заметки станут очень большими, можно подумать о виртуализации read-only режима. Для editor-режима это сложнее, потому что BlockNote сам управляет ProseMirror-документом.


24.5. Более строгая модель данных

Сейчас `editorContent` типизирован мягко, потому что BlockNote-блоки сложные и меняются между версиями.

Можно постепенно сделать:

- собственные typed wrappers;
- schema version migrations;
- отдельные типы для props каждого кастомного блока.


25. Краткий итог
----------------

Модуль заметок сейчас — это гибрид:

- BlockNote отвечает за сложное редактирование;
- MyMind отвечает за UX, дизайн, right sidebar, режим чтения, хранение, миграции и кастомные блоки;
- Electron отвечает за локальные JSON и assets;
- React связывает всё в единый интерфейс.

Самое важное текущее архитектурное решение:

Не хранить тяжёлые файлы внутри `editorContent`.

Для текста, таблиц, списков, markdown и drawing JSON подходит хорошо.

Для видео, аудио, файлов и крупных изображений лучше хранить только ссылку на локальный asset, а сам файл держать отдельно.

Именно это делает редактор отзывчивым и сохраняет идею локального приложения без сервера.
